DIARY ENTRY: DATA ACQUISITION II — DATA STORAGE AND DATABASE SCHEMA

PROJECT CONTEXT

At this stage of the project, the focus moved from data collection to organizing the acquired cricket data into a structured, reproducible system. The goal of the project is to study how match context and pressure situations (league vs playoffs vs finals) are associated with performance variability and risk in IPL players, including both batters and bowlers.

To support this objective, the project uses a relational database design that clearly represents relationships between matches, players, and match-level performance outcomes.

⸻

DATABASE SYSTEM

A SQLite relational database was selected for this project. SQLite is lightweight, portable, and well-suited for analytical workflows where multiple related tables must be queried together. Using a single .db file ensures the analysis is reproducible and easy to share.

⸻

STRUCTURED CRICKET TABLES

TABLE: matches
Primary Key: id

This table stores match-level metadata and provides the contextual anchor for the entire analysis.

Key fields include:
	•	season
	•	date
	•	venue
	•	match_type
	•	team1, team2
	•	winner and result fields
	•	stage_bucket (league / playoffs / final)

The stage_bucket variable is central to the project, as it operationalizes pressure through match importance rather than subjective interpretation.

⸻

TABLE: players
This table stores standardized player identifiers used across all performance records. It ensures consistent naming and avoids duplication when aggregating player-level statistics.

⸻

TABLE: player_match
This table stores player-level performance outcomes aggregated at the match level.

Key fields include:
	•	match_id (foreign key → matches.id)
	•	player
	•	runs
	•	balls
	•	dismissal indicator
	•	strike_rate
	•	derived indicators such as low-score outcomes

This table is the primary source for analyzing:
	•	performance variability
	•	risk of failure (e.g., low scores)
	•	differences between league and high-pressure matches

It enables both match-level and player-level analysis.

⸻

RELATIONAL DESIGN AND JOIN STRATEGY

The schema follows a clear relational structure: matches  →  player_match  →  players

This design allows:
	•	aggregation of player performance within matches
	•	comparison of player behavior across match stages
	•	extension to future analyses involving bowling metrics or team-level features

Foreign key relationships ensure referential integrity and make joins explicit and interpretable.

⸻

HANDLING DERIVED FEATURES

Derived variables such as:
	•	low-score indicators
	•	performance volatility
	•	match-level averages

are not hard-coded into the schema, but computed through SQL queries or pandas pipelines. This keeps the database normalized while preserving analytical flexibility.

⸻

REFLECTION

Designing the database schema clarified how match context and player performance interact across different pressure situations. Separating match metadata from player-level outcomes ensures the analysis remains interpretable and extensible.

This database structure now provides a solid foundation for:
	•	exploratory data analysis
	•	pressure-aware performance metrics
	•	future machine learning models predicting performance risk under high-stakes conditions
