DIARY ENTRY: DATA ACQUISITION II — DATA STORAGE AND DATABASE SCHEMA

PROJECT CONTEXT

At this stage of the project, the focus moved from data collection to organizing the acquired cricket data into a structured and reproducible system. The objective of the project is to study how match context and competitive pressure—operationalized through tournament stage (league, playoffs, finals)—are associated with performance variability and risk in IPL players.

Importantly, the project considers both batting and bowling performance, allowing performance risk to be analyzed across different player roles rather than being limited to run-scoring outcomes alone.

To support this objective, a relational database schema was designed to clearly represent relationships between matches, players, and match-level performance outcomes.

⸻

DATABASE SYSTEM

A SQLite relational database was selected for this project. SQLite is lightweight, portable, and well-suited for analytical workflows where multiple related tables must be queried together. Using a single .db file ensures the analysis is reproducible, transparent, and easy to share across environments.

⸻

STRUCTURED CRICKET TABLES

TABLE: matches
Primary Key: id

This table stores match-level metadata and provides the contextual anchor for the entire analysis.

Key fields include:
	•	season
	•	date
	•	venue
	•	match_type
	•	team1, team2
	•	winner and result fields
	•	stage_bucket (league / playoffs / final)

The stage_bucket variable is central to the project, as it operationalizes competitive pressure using match importance rather than subjective or external indicators.

⸻

TABLE: players
This table stores standardized player identifiers used across all performance records. It ensures consistent naming, avoids duplication, and supports analysis across different player roles (batters, bowlers, and all-rounders).

⸻

TABLE: player_match
This table stores player-level performance outcomes aggregated at the match level, with one row per player per match.

Key fields include:
	•	match_id (foreign key → matches.id)
	•	player
	•	Batting metrics:
	•	runs
	•	balls faced
	•	dismissal indicator
	•	strike rate
	•	Bowling metrics (where applicable):
	•	overs bowled
	•	runs conceded
	•	wickets taken
	•	economy rate
	•	Derived indicators such as low-score outcomes

This table is the primary source for analyzing:
	•	performance variability
	•	risk of failure (e.g., low scores for batters, high economy or wicketless spells for bowlers)
	•	differences between league and high-pressure matches

It supports both match-level aggregation and player-level longitudinal analysis.

⸻

RELATIONAL DESIGN AND JOIN STRATEGY

The schema follows a clear relational structure:
matches  →  player_match  →  players

This design enables:
	•	aggregation of player performance within matches
	•	comparison of player behavior across tournament stages
	•	unified analysis of batting and bowling performance under pressure
	•	straightforward extension to team-level or role-specific analyses

Foreign key relationships ensure referential integrity and make joins explicit and interpretable.

⸻

HANDLING DERIVED FEATURES

Derived variables such as:
	•	low-score indicators
	•	batting and bowling performance volatility
	•	match-level averages and risk rates

are computed dynamically using SQL queries or pandas pipelines rather than being permanently stored in the schema. This keeps the database normalized while preserving flexibility for exploratory analysis and future modeling.

⸻

REFLECTION

Designing the database schema clarified how match context and player performance interact across different pressure situations. By separating match metadata from player-level outcomes and including both batting and bowling metrics in a unified structure, the schema supports a comprehensive and interpretable analysis of performance risk.

This database structure now provides a strong foundation for:
	•	exploratory data analysis
	•	pressure-aware batting and bowling metrics
	•	future machine learning models focused on performance variability under high-stakes match conditions
