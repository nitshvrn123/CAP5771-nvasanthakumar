DIARY ENTRY: DATA ACQUISITION II — DATA STORAGE AND DATABASE SCHEMA (REVISED)

PROJECT CONTEXT

At this stage of the project, I moved from data collection to organizing the IPL cricket data into a structured and reproducible database. My goal is to study how match context and competitive pressure—operationalized using tournament stage (league, playoffs, final)—relates to performance risk and variability for IPL players.

A key design decision is to treat batting and bowling as distinct analytical roles, since the performance signals and risk mechanisms are different for each role. To support this, I used separate batting and bowling performance tables linked to the same match context.

⸻

DATABASE SYSTEM

I used a SQLite relational database stored as a single file : 

ipl_cricket_only.db

SQLite is lightweight, portable, and works well for analytical workflows where I need to join match metadata with player-level performance records. Using one database file helps keep the project reproducible and easy to run in new environments (e.g., Colab).

⸻

STRUCTURED TABLES (CURRENT SCHEMA)

1) TABLE: matches
Primary Key: id

Purpose:
Stores the match-level metadata and provides the “context anchor” for all analysis.

Key columns include:
- id
- season
- date
- venue
- match_type
- team1, team2
- winner, result
- stage_bucket (league / playoffs / final)

The stage_bucket field is the project’s pressure proxy, derived from tournament stage.

2) TABLE: players

Purpose:
Stores the standardized player identifier list used across the performance tables.

Key columns include:
- player_name

(Practically, player_name is treated as a stable identifier across tables.)

3) TABLE: player_match_batting

Purpose:
Stores batting-only performance at the player–match level (one row per batter per match).

Key columns include:
- match_id (FK → matches.id)
- player (FK → players.player_name)
- runs
- balls
- strike_rate
- low_score (derived: runs < 10)
- season, date, venue, match_type, stage_bucket (included for easier analysis)

Uniqueness:
No explicit primary key is enforced in SQLite; in practice each record is identified by (match_id, player).

4) TABLE: player_match_bowling

Purpose:
Stores bowling-only performance at the player–match level (one row per bowler per match).

Key columns include:
- match_id (FK → matches.id)
- bowler (FK → players.player_name)
- balls
- overs
- runs_conceded
- wickets
- wides, no_balls
- dot_balls, dot_ball_rate
- economy
- strike_rate_balls_per_wicket (can be missing when wickets = 0)
- season, date, venue, match_type, stage_bucket (included for easier analysis)

Uniqueness:
No explicit primary key is enforced in SQLite; in practice each record is identified by (match_id, bowler).

⸻

RELATIONAL DESIGN AND JOIN STRATEGY

The database follows this join structure:

matches.id  →  player_match_batting.match_id
matches.id  →  player_match_bowling.match_id

players.player_name  →  player_match_batting.player
players.player_name  →  player_match_bowling.bowler

This design keeps match context in one place (matches) while maintaining separate, role-specific outcome tables for batting and bowling. It prevents mixing incompatible performance signals and makes stage-aware analysis straightforward.

⸻

HANDLING DERIVED FEATURES

I store “atomic” performance columns in the database (runs, balls, wickets, economy, etc.). I do not permanently store most derived risk metrics to avoid duplication and keep the database flexible.

Derived features are computed in the notebook using SQL + pandas, including:
- low-score rates
- performance variance by stage
- pressure impact indices (batting and bowling)

When building summary indices, I clip improvements under pressure at zero so that:
- 0 represents stability (not missingness)
- positive values represent worse performance under pressure

⸻

REFLECTION

Designing this schema clarified the relationship between match context and player outcomes. Separating batting and bowling into role-specific player–match tables improved interpretability and made my downstream analysis cleaner. This database now provides a strong foundation for exploratory analysis, risk metrics by stage, and future machine learning extensions (predicting expected runs or economy under similar match conditions).
