DIARY ENTRY: DATA STORAGE AND DATABASE SCHEMA

PROJECT CONTEXT

At this stage, the project transitioned from raw data acquisition to structured data organization. The goal was to design a database schema that cleanly represents cricket match context and player performance outcomes in a way that supports reproducible analysis of performance variability under pressure.

Because the project focuses exclusively on cricket analytics, the database schema was designed around match-level context and player-level performance, with explicit support for comparisons across different match stages (league, playoffs, finals).

⸻

DATABASE CHOICE

A SQLite relational database was used to store all project data in a single .db file.

SQLite was chosen because it is:
	•	Lightweight and portable
	•	Easy to reproduce across systems
	•	Well-suited for analytical workflows involving multiple related tables
	•	Fully compatible with SQL-based joins and pandas-based analysis

This design ensures that all analysis can be reproduced directly from the database without relying on external services.

⸻

1) CORE CRICKET TABLES

Table: matches
Primary key: id

This table stores match-level metadata for each IPL match, including:
	•	Season
	•	Date
	•	City and venue
	•	Teams involved
	•	Toss and result information

A key derived column in this table is:
	•	stage_bucket
Categorizes matches into:
	•	league
	•	playoffs
	•	final

This variable is central to the project, as it operationalizes match pressure using competition stage rather than subjective or external indicators.

The matches table serves as the anchor table for all joins in the database.

⸻

Table: players
This table stores unique player identifiers used consistently across performance tables. It provides a clean reference list of players appearing in the dataset.

⸻

Table: player_match
This table stores player-level performance aggregated at the match level, with one row per player per match.

Key fields include:
	•	Match identifier (match_id)
	•	Player name
	•	Runs scored
	•	Balls faced
	•	Strike rate
	•	Dismissal indicators
	•	Derived performance flags (e.g., low-score outcomes)

This table is the foundation for:
	•	Player performance summaries
	•	Risk and variability calculations
	•	Comparisons between league and high-pressure matches

The table is joined to matches using match_id to incorporate match context.

⸻

DATABASE RELATIONSHIPS

The core relational structure of the database is:matches (id)
   ↓
player_match (match_id)

This design enables:
	•	Match-level aggregation of player outcomes
	•	Player-level analysis across different match stages
	•	Clear separation between match context and player performance

⸻

HANDLING DERIVED FEATURES

Rather than storing every derived statistic directly in the database, the schema stores clean, atomic performance fields (runs, balls, strike rate, dismissals). Derived metrics such as:
	•	Low-score rates
	•	Performance variance
	•	Pressure impact indices

are computed dynamically using SQL queries and pandas during analysis. This approach avoids data duplication and keeps the database flexible for future extensions.

⸻

REFLECTION

The most important design decision was to center the schema around match stage as a pressure proxy, using only cricket-intrinsic data. By structuring the database around matches and player_match, the schema cleanly supports reproducible, interpretable analysis of how player performance changes across competition stages.

This design ensures that all conclusions drawn in later stages are grounded in objective match outcomes rather than subjective or external signals, making the project analytically robust and defensible.
