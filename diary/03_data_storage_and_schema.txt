DIARY ENTRY: DATA STORAGE AND DATABASE SCHEMA

PROJECT CONTEXT

At this stage, the project transitioned from raw data acquisition to structured data organization. The goal was to design a database schema that cleanly represents cricket match context and both batting and bowling performance outcomes in a way that supports reproducible analysis of performance variability under pressure.

Because the project focuses exclusively on cricket analytics, the database schema was designed around match-level context and player-level performance, with explicit support for comparisons across different match stages (league, playoffs, finals) and player roles.

⸻

DATABASE CHOICE

A SQLite relational database was used to store all project data in a single .db file.

SQLite was chosen because it is:
	•	Lightweight and portable
	•	Easy to reproduce across systems
	•	Well-suited for analytical workflows involving multiple related tables
	•	Fully compatible with SQL-based joins and pandas-based analysis

This design ensures that all analysis can be reproduced directly from the database without relying on external services.

⸻

1) CORE CRICKET TABLES

Table: matches
Primary key: id

This table stores match-level metadata for each IPL match, including:
	•	Season
	•	Date
	•	City and venue
	•	Teams involved
	•	Toss and result information

A key derived column in this table is:
	•	stage_bucket, which categorizes matches into:
	•	league
	•	playoffs
	•	final

This variable operationalizes competitive pressure using tournament structure and is central to the project’s analysis.

The matches table serves as the anchor table for all joins in the database.

⸻

Table: players
This table stores unique player identifiers used consistently across performance tables. It provides a clean reference list of players appearing in the dataset and supports analysis across different player roles (batters, bowlers, all-rounders).

⸻

Table: player_match
This table stores player-level performance aggregated at the match level, with one row per player per match.

Key fields include:
	•	Match identifier (match_id)
	•	Player name
	•	Batting metrics:
	•	Runs scored
	•	Balls faced
	•	Strike rate
	•	Dismissal indicator
	•	Bowling metrics (where applicable):
	•	Overs bowled
	•	Runs conceded
	•	Wickets taken
	•	Economy rate

Derived performance flags (e.g., low-score outcomes) are also included where relevant.

This table is the foundation for:
	•	Batting and bowling performance summaries
	•	Risk and variability calculations
	•	Comparisons between league and high-pressure matches
	•	Role-agnostic analysis of player performance under pressure

The table is joined to matches using match_id to incorporate match context.

⸻

DATABASE RELATIONSHIPS

The core relational structure of the database is:
matches (id)
   ↓
player_match (match_id)

This design enables:
	•	Match-level aggregation of player outcomes
	•	Player-level analysis across different match stages
	•	Unified analysis of batting and bowling performance within the same framework
	•	Clear separation between match context and player performance

⸻

HANDLING DERIVED FEATURES

Rather than storing every derived statistic directly in the database, the schema stores clean, atomic performance fields (e.g., runs, balls, wickets, runs conceded). Derived metrics such as:
	•	Low-score rates
	•	Bowling economy volatility
	•	Performance variance
	•	Pressure impact indices

are computed dynamically using SQL queries and pandas during analysis. This approach avoids data duplication and keeps the database flexible for future extensions.

⸻

REFLECTION

The most important design decision was to center the schema around match stage as a pressure proxy, using only cricket-intrinsic data. By structuring the database around matches and a unified player_match table that includes both batting and bowling metrics, the schema supports robust, reproducible analysis of how different types of player performance change across competition stages.
