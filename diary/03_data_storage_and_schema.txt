DIARY ENTRY: DATA STORAGE AND DATABASE SCHEMA

PROJECT CONTEXT:
At this stage, the project moved from data collection to organizing everything into a structured, reproducible system. Because the project combines structured IPL match/player performance data with unstructured YouTube fan sentiment, I designed a relational schema that supports joining these sources for analysis.

⸻

DATABASE CHOICE

I used a SQLite relational database (single .db file). SQLite is lightweight, portable, and ideal for analytical workflows where multiple related tables must be queried together.

⸻

1) Structured cricket tables

Table: matches

Key column: id
Stores match-level metadata such as season, city, date, venue, teams (team1, team2), toss details, winner/result fields, and a derived stage label.

Important column for pressure analysis:
	•	stage_bucket → categorizes match importance (league vs playoffs vs final)

Helper column used for linking YouTube to matches:
	•	_team_text → a derived text field combining team identifiers to help detect team mentions during linking.

This table is the anchor table for all joins.

⸻

Table: players

Stores player identifiers used across match-level performance outputs.

⸻

Table: player_match

Stores player-level performance aggregated at the match level (player outcomes per match).
This table is used to compute performance risk and variability metrics and will be joined to match context (matches) using match identifiers.

⸻

2) YouTube media tables

Table: youtube_videos

Key column: video_id
Contains metadata for each YouTube video used in the fan pressure proxy, including title/channel, publication timestamp, the query used to retrieve it, and inferred labels like pressure stage/team mentions.

⸻

Table: youtube_comments

Contains individual fan comments and NLP-derived fields such as sentiment score and sentiment bin, along with engagement signals (e.g., like count).
Each comment is associated with a video via video_id.

⸻

3) Linking table (critical join)

Table: match_media_links

This is the key table that connects cricket data with YouTube data.

Fields include:
	•	match_id → links to matches.id
	•	video_id → links to youtube_videos.video_id
	•	link_method → how the match/video was linked (date window, stage match, team match)
	•	link_confidence → confidence score for the linkage

This table enables the full join chain:

youtube_comments → youtube_videos → match_media_links → matches → player_match

⸻

HANDLING UNSTRUCTURED DATA

Fan comment text is stored in raw form along with computed sentiment fields so that pressure indicators can be built without re-running NLP repeatedly. Aggregated pressure features (comment volume, mean sentiment, sentiment volatility) can be derived per match or stage using SQL queries.

⸻

REFLECTION

The most important design choice was adding match_media_links, because it makes the connection between fan reactions and match context explicit instead of assuming every “final reaction” video maps cleanly to a single match. This improves interpretability and supports future extensions (better linking rules, more linked videos, higher confidence thresholds).
