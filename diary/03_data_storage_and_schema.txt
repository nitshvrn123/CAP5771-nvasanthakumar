DIARY ENTRY: DATA STORAGE AND DATABASE SCHEMA

PROJECT CONTEXT

At this stage, the project transitioned from raw data acquisition to structured data organization. The goal was to design a database schema that cleanly represents cricket match context and role-specific player performance outcomes, enabling reproducible analysis of performance variability under competitive pressure.

Because the project focuses exclusively on cricket analytics, the database schema was designed around match-level context and separate batting and bowling performance tables, with explicit support for comparisons across match stages (league, playoffs, finals) and player roles.

⸻

DATABASE CHOICE

A SQLite relational database was used to store all project data in a single .db file (ipl_cricket_only.db).

SQLite was chosen because it is:
	•	lightweight and portable
	•	easy to reproduce across systems
	•	well-suited for analytical workflows involving multiple related tables
	•	fully compatible with SQL-based joins and pandas-based analysis

The SQLite database file is included in the repository, ensuring that all analysis can be reproduced directly from stored data without external dependencies.

⸻

CORE CRICKET TABLES

Table: matches
Primary Key: id

This table stores match-level metadata for each IPL match, including:
	•	season
	•	date
	•	venue
	•	team1, team2
	•	match_type
	•	winner and result information

A key derived column in this table is:
	•	stage_bucket, which categorizes matches into:
	•	league
	•	playoffs
	•	final

This variable operationalizes competitive pressure using tournament structure and serves as the contextual anchor for all analysis.

⸻

Table: players
This table stores standardized player identifiers used consistently across all performance tables. It provides a reference list of players appearing in the dataset and supports analysis across different player roles (batters, bowlers, and all-rounders).

⸻

Table: player_match_batting
This table stores batting-only performance outcomes aggregated at the match level, with one row per batter per match.

Key fields include:
	•	match_id (foreign key → matches.id)
	•	player
	•	runs
	•	balls
	•	strike_rate
	•	low_score indicator
	•	embedded match context variables (season, venue, match_type, stage_bucket)

No explicit primary key constraint is enforced in SQLite. However, each record is uniquely identified in practice by the composite of (match_id, player).

This table supports:
	•	batting performance variability analysis
	•	low-score risk estimation
	•	stage-specific comparisons across league, playoffs, and finals

⸻

Table: player_match_bowling
This table stores bowling-only performance outcomes aggregated at the match level, with one row per bowler per match.

Key fields include:
	•	match_id (foreign key → matches.id)
	•	bowler
	•	overs
	•	runs_conceded
	•	wickets
	•	economy
	•	dot_ball_rate
	•	strike_rate_balls_per_wicket
	•	embedded match context variables (season, venue, match_type, stage_bucket)

No explicit primary key constraint is enforced in SQLite. However, each record is uniquely identified in practice by the composite of (match_id, bowler).

This table supports:
	•	bowling performance variability analysis
	•	identification of high-cost spells
	•	stage-specific changes in economy and wicket-taking effectiveness
