DATA ACQUISITION PART II — DATA DICTIONARY (Version 1)

Project: Quantifying Performance Risk in Professional Cricket Across Match Stages

This data dictionary defines the core variables stored in the relational database for Milestone 1.
All data is derived exclusively from structured cricket match and player performance records.

The goal of this database is to support analysis of performance variability and risk for both batters and bowlers under different match pressure stages (league, playoffs, final).

⸻

TABLE: matches

id (INTEGER)
	•	Unique identifier for each IPL match.
	•	Primary Key.

date (DATE)
	•	Date on which the match was played.

season (INTEGER)
	•	Season/year extracted from the match date.

venue (TEXT)
	•	Stadium or city where the match was played.

team1 (TEXT)
	•	First competing team.

team2 (TEXT)
	•	Second competing team.

match_type (TEXT)
	•	Match format/category (e.g., league match, qualifier, eliminator, final).

stage_bucket (TEXT)
	•	Derived match stage used as an objective pressure proxy:
	•	league
	•	playoffs
	•	final

winner (TEXT)
	•	Winning team.

result (TEXT)
	•	Match outcome type (runs, wickets, tie, no result).

⸻

TABLE: players

player_name (TEXT)
	•	Standardized player name used consistently across all performance records.

Note: No explicit primary key is enforced in SQLite; player names act as stable identifiers in practice.

This table serves as a reference table for all player roles, including:
	•	Specialist batters
	•	Specialist bowlers
	•	All-rounders

⸻

TABLE: player_match_batting

This table stores batting-only player–match performance outcomes
(one row per batter per match).

match_id (INTEGER)
	•	Match identifier linking performance to match context.
	•	Foreign key → matches.id.

player (TEXT)
	•	Batter name.
	•	Foreign key → players.player_name.

runs (INTEGER)
	•	Runs scored by the player in the match.

balls (INTEGER)
	•	Balls faced by the player.

strike_rate (REAL)
	•	Batting strike rate: (runs / balls) × 100.

low_score (INTEGER: 0/1)
	•	Derived indicator:
	•	1 if runs < 10
	•	0 otherwise
	•	Used as a proxy for batting failure risk.

stage_bucket (TEXT)
	•	Match stage category inherited from matches.

Primary Key: None explicitly defined (SQLite).
Each row represents a unique player–match batting record in practice.

⸻

TABLE: player_match_bowling

This table stores bowling-only player–match performance outcomes
(one row per bowler per match).

match_id (INTEGER)
	•	Match identifier linking performance to match context.
	•	Foreign key → matches.id.

bowler (TEXT)
	•	Bowler name.
	•	Foreign key → players.player_name.

balls (INTEGER)
	•	Balls delivered by the bowler.

overs (REAL)
	•	Overs bowled (balls / 6).

runs_conceded (INTEGER)
	•	Runs conceded while bowling.

wickets (INTEGER)
	•	Number of wickets taken.

economy (REAL)
	•	Runs conceded per over.

dot_ball_rate (REAL)
	•	Proportion of dot balls delivered.

strike_rate_balls_per_wicket (REAL)
	•	Bowling strike rate measured as balls per wicket.

stage_bucket (TEXT)
	•	Match stage category inherited from matches.

Primary Key: None explicitly defined (SQLite).
Each row represents a unique bowler–match spell, identified in practice by
(match_id, bowler).

⸻

DERIVED ANALYSIS FEATURES

(Computed during EDA; not stored as raw tables)

Batting risk metrics
	•	avg_runs (by stage)
	•	low_score_rate
	•	run_variance
	•	pressure_impact_index (batting)

Bowling risk metrics
	•	avg_economy
	•	economy_variance
	•	high_cost_spell_rate
	•	pressure_impact_index (bowling)

Labels

pressure_label
	•	Stable
	•	More risky under pressure
	•	High pressure drop

Note: Improvements under pressure are clipped at zero in summary indices;
a value of zero represents stability, not missing data.

⸻

NOTES AND SCOPE
	•	Milestone 1 focuses on:
	•	data acquisition
	•	database design
	•	exploration readiness
	•	Batting and bowling outcomes are stored in separate tables to preserve role-specific interpretation.
	•	All variables are derived from objective match outcomes, not psychological inference.
	•	Later milestones may:
	•	refine role-specific pressure metrics
	•	introduce uncertainty quantification
	•	extend analysis to team-level dynamics

⸻

SUMMARY

This data dictionary defines a cricket-only, role-aware relational schema that supports exploratory analysis of performance risk for both batters and bowlers across match stages.
It provides a defensible and reproducible foundation for advanced analytics without relying on external or subjective data sources.
