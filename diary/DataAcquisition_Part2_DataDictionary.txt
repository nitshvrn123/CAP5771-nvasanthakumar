DATA ACQUISITION PART II — DATA DICTIONARY (Version 2)


This data dictionary defines the variables stored in the SQLite database
(ipl_cricket_only.db) for Milestone 1.

All data is derived exclusively from structured IPL match and player performance records.
No external or psychological variables are used.

The database supports stage-aware analysis of performance variability and risk
for both batters and bowlers across three match stages:
league, playoffs, and final

⸻

TABLE: matches

id (INTEGER)
- Unique identifier for each IPL match.
- Primary Key.

season (TEXT)
- IPL season (e.g., "2007/08").

date (TEXT)
- Match date.

city (TEXT)
- City where match was played (may contain null values).

venue (TEXT)
- Stadium or venue name.

team1 (TEXT)
- First competing team.

team2 (TEXT)
- Second competing team.

winner (TEXT)
- Winning team.

result (TEXT)
- Match result type (runs, wickets, tie, no result).

match_type (TEXT)
- Match classification (league, qualifier, eliminator, final).

stage_bucket (TEXT)
- Derived tournament stage:
  - league
  - playoffs
  - final
- Used as the objective pressure proxy.

⸻

TABLE: players

player_name (TEXT)
- Standardized player name used across performance tables.
- Acts as a stable identifier in practice.

⸻

TABLE: player_match_batting

One row per batter per match.

match_id (INTEGER)
- Foreign Key → matches.id

player (TEXT)
- Foreign Key → players.player_name

runs (INTEGER)
- Runs scored in the match.

balls (INTEGER)
- Balls faced.

strike_rate (REAL)
- (runs / balls) × 100

low_score (INTEGER: 0/1)
- Derived variable:
  - 1 if runs < 10
  - 0 otherwise
- Proxy for batting failure risk.

season (TEXT)
date (TEXT)
venue (TEXT)
match_type (TEXT)
stage_bucket (TEXT)

These contextual variables are retained for stage-aware analysis.

Primary Key:
- Not explicitly enforced.
- Each record is uniquely identified in practice by (match_id, player).

⸻

TABLE: player_match_bowling

One row per bowler per match.

match_id (INTEGER)
- Foreign Key → matches.id

bowler (TEXT)
- Foreign Key → players.player_name

balls (INTEGER)
- Balls delivered.

overs (REAL)
- Overs bowled.

runs_conceded (INTEGER)
- Runs conceded.

wickets (INTEGER)
- Wickets taken.

wides (INTEGER)
no_balls (INTEGER)

dot_balls (INTEGER)
dot_ball_rate (REAL)

economy (REAL)
- Runs conceded per over.

strike_rate_balls_per_wicket (REAL)
- Balls per wicket.
- May be null when wickets = 0.

season (TEXT)
date (TEXT)
venue (TEXT)
match_type (TEXT)
stage_bucket (TEXT)

Primary Key:
- Not explicitly enforced.
- Each record is uniquely identified in practice by (match_id, bowler).

⸻

DERIVED ANALYSIS FEATURES (Computed in Notebook)

The following metrics are not stored as raw tables but computed dynamically:

Batting:
- Average runs by stage
- Low-score rate
- Variance of runs
- Pressure Impact Index (batting)

Bowling:
- Average economy by stage
- High-cost spell rate
- Variance of economy
- Pressure Impact Index (bowling)

Pressure Labels:
- Stable
- More risky under pressure
- High pressure drop

When computing impact indices, improvements under pressure are clipped at zero.
A value of zero represents stability (not missing data).

⸻

SCOPE

Milestone 1 focuses on:
- Data acquisition
- Database schema design
- Exploratory analysis
- Risk metric construction

The schema is designed to support future machine learning models that predict:
- Runs scored (batting model)
- Economy rate (bowling model)
under similar match conditions.

⸻

SUMMARY

This data dictionary documents a structured, role-aware relational schema
that supports reproducible and interpretable analysis of IPL performance
risk across match stages.
